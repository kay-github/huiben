# 回本计算器项目工作流

- Project: 回本计算器
- Purpose: 抽象本项目从产品定义到上线的系统化工作流
- Scope: 适用于小功能、强约束、强调高质量交付的 AI 协作项目
- Last Updated: 2026-03-29

## 1. 文档目的

这份文档不是介绍产品功能本身，而是抽象出本项目的**完整工作流**，让后来参与的人快速理解：

1. 这个项目的技术实现本身并不复杂
2. 真正决定结果质量的，不是写代码速度，而是：
   - 需求收敛
   - 角色分工
   - 约束固化
   - 交付物溯源
   - 多轮验收
   - 闭环修正
3. 这套方法可以复用到类似的“小而美单功能产品”中

## 2. 项目本质

回本计算器本质上是一个**单功能、单页面、轻逻辑**的产品：

1. 输入亏损幅度
2. 输出回本涨幅
3. 不做功能扩展

从开发角度看，它不难。

从项目管理角度看，真正的难点在于：

1. 防止需求发散
2. 防止设计偏题
3. 防止开发擅自扩展
4. 防止测试标准漂移
5. 防止不同 AI 角色之间的理解失真

因此，本项目采用的是一种**以文档约束和验收闸门为核心的 AI 协作工作流**。

## 4. 工作流总览

整个项目按以下阶段推进：

1. 产品定义阶段
2. 规则固化阶段
3. 设计阶段
4. 开发阶段
5. 开发验收阶段
6. 网页展示版转换阶段
7. 测试闸门阶段
8. 上线与归档阶段

核心逻辑不是“一次性交付”，而是：

**先定义，再约束，再执行，再验收，再循环修正，直到通过。**

## 5. 角色分工

### 5.1 产品经理角色

职责：

1. 收敛需求
2. 明确边界
3. 输出可执行文档
4. 作为最终验收裁判
5. 控制设计、开发、测试不偏航

产出物：

1. `product-design/01-prd.md`
2. `product-design/02-test-cases.md`
3. `product-design/03-development-rules.md`

### 5.2 UI 设计 AI

职责：

1. 根据 PRD 和产品边界设计界面
2. 不扩展功能
3. 只在既定风格内迭代

产出物：

1. `product-design/ui-design-prompt/*.md`
2. `product-design/stitch/`

说明：

UI 设计不是自由创作，而是被产品边界和验收结论逐轮收敛。

### 5.3 开发 AI

职责：

1. 严格按 PRD、测试用例、开发规则实现
2. 优先最小正确方案
3. 不擅自新增功能
4. 在验收反馈下做定向修正

产出物：

1. `huiben-calculator/`
2. `huiben-calculator-web/`
3. `product-design/07-development-ai-prompt.md`
4. `product-design/08-development-ai-followup-prompt.md`
5. `product-design/09-web-development-ai-prompt.md`

### 5.4 测试 AI

职责：

1. 按测试标准做验证
2. 给出通过 / 不通过结论
3. 不改代码
4. 只输出问题和复现信息

产出物：

1. `product-design/10-web-test-ai-prompt.md`
2. 测试结论本身

### 5.5 人工决策者

职责：

1. 确认产品方向
2. 决定阶段切换
3. 审阅最终结果
4. 决定是否进入展示、上线或下一轮修正

## 6. 关键交付物体系

## 7. 标准工作流

以下是本项目的标准执行流程。

### 阶段 1：产品定义

目标：把需求从“想法”变成“可执行边界”。

动作：

1. 明确产品只做什么
2. 明确不做什么
3. 明确卖点
4. 明确视觉目标
5. 明确交互规则
6. 明确公式和异常处理

输出：

1. PRD
2. 测试用例
3. 开发规则

进入下一阶段的条件：

1. 产品范围已被锁定
2. 关键计算逻辑已明确
3. 验收标准已形成文档

### 阶段 2：设计收敛

目标：得到可以进入开发的视觉基线。

动作：

1. 向 UI 设计 AI 提供有边界的提示词
2. 审查设计稿是否偏离产品定义
3. 如果偏离，输出下一轮定向修改提示词
4. 重复，直到达到“可开发定稿”标准

关键原则：

1. 不让 UI 自由扩展功能
2. 不让设计只追求酷炫而牺牲产品目标
3. 不用“大概差不多”作为开发输入

进入下一阶段的条件：

1. 单页面结构稳定
2. 视觉重心正确
3. 无越界模块
4. 可作为开发还原依据

### 阶段 3：开发实现

目标：将定稿设计和产品规则转化为可运行实现。

动作：

1. 向开发 AI 明确必须遵守的文件
2. 明确冲突优先级
3. 明确实现目录与技术栈
4. 明确验证要求

关键原则：

1. 实现必须受文档约束
2. 先完成最小正确版本
3. 不做过度工程化
4. 不擅自扩展产品边界

进入下一阶段的条件：

1. 基础逻辑完成
2. 页面结构完成
3. 主要状态完成

### 阶段 4：产品经理验收

目标：判断开发实现是否符合产品定义，而不是只看“能不能运行”。

动作：

1. 检查实现文件
2. 对照 PRD 和测试用例逐项验收
3. 找出偏差
4. 若有问题，形成定向修正提示词给开发 AI

关键原则：

1. 不全盘推翻
2. 只修当前真正影响验收的偏差
3. 让反馈具体到文件和问题点

进入下一阶段的条件：

1. 开发实现满足当前阶段目标
2. 没有中高优先级产品偏差

### 阶段 5：网页展示版转换

目标：在不改产品本质的前提下，快速产出一个浏览器可运行版本用于演示。

动作：

1. 以已验收小程序版作为行为基线
2. 在独立目录生成网页版
3. 复用逻辑、文案、状态和视觉方向
4. 不改乱原小程序项目

关键原则：

1. 平台转换，不是重新开发一个新产品
2. 网页版继承已验收结果
3. 优先静态可运行方案

进入下一阶段的条件：

1. 网页版可直接打开运行
2. 核心行为与小程序基线一致

### 阶段 6：测试闸门

目标：通过测试确认版本是否可视为上线版本。

动作：

1. 向测试 AI 提供明确测试提示词
2. 只允许测试 AI 输出通过 / 不通过
3. 若不通过，问题回流给开发 AI 修复
4. 修复后再次测试

关键原则：

1. 测试不写代码
2. 开发不自证通过
3. 不通过就继续循环
4. 直到测试明确给出通过结论

这一步形成项目的**最后质量闸门**。

### 阶段 7：上线与归档

目标：让项目从“完成开发”变成“可复用经验”。

动作：

1. 记录最终通过结论
2. 保留关键提示词和验收结果
3. 形成工作流文档
4. 为后续项目复用方法论

## 8. 核心控制机制

本项目之所以能稳定推进，不在于任何单个 AI 更强，而在于以下控制机制。

### 8.1 文档先行

先写规则，再做设计和开发。

作用：

1. 避免边做边变
2. 避免口头约束丢失
3. 让不同 AI 使用同一套基准

### 8.2 角色隔离

每个角色只做自己该做的事：

1. 产品负责定义和验收
2. 设计负责输出视觉方案
3. 开发负责实现
4. 测试负责放行

作用：

1. 降低角色混乱
2. 降低“自己定义、自己实现、自己通过”的风险

### 8.3 提示词化执行

不是简单地下达口头任务，而是把每一次执行都做成有约束的提示词文档。

作用：

1. 保证交接清晰
2. 便于追溯
3. 便于复用

### 8.4 多轮验收收敛

设计不是一次定稿，开发也不是一次通过。

流程是：

1. 先出一个版本
2. 验收指出偏差
3. 定向修正
4. 再验收
5. 直到通过

作用：

1. 防止大返工
2. 防止问题堆积到最后

### 8.5 独立测试闸门

只要测试 AI 没有明确说“通过”，就不能视为完成。

作用：

1. 让“完成”有客观标准
2. 防止开发阶段自我感觉良好

## 9. 本项目可复用的方法论

这套流程适合以下类型项目：

1. 功能单一
2. 技术实现不重
3. 对体验质量要求高
4. 多 AI 角色协作
5. 容易因范围发散而失控

可复用模板如下：

1. 先写 PRD
2. 再写测试用例
3. 再写开发规则
4. 再让设计出稿
5. 产品经理做设计审查
6. 设计通过后，给开发 AI 实现提示词
7. 开发完成后，产品经理做产品验收
8. 若有问题，给开发 AI 定向修正提示词
9. 必要时做平台转换版本
10. 最后由测试 AI 做放行
11. 测试不通过则修复并重测
12. 测试通过后归档全部关键文档

## 10. 为什么这个项目成功

这个项目成功的关键，不是因为“公式简单”。

真正关键的是：

1. 从一开始就锁住了产品边界
2. 每个角色都拿到了明确约束
3. 每一步都有输入和输出
4. 每一轮都有可追溯文档
5. 所有通过结论都建立在验收之上，而不是印象之上

换句话说：

**这个项目本身不难，难的是把一个简单项目做得稳定、统一、可控、可验证。**

## 11. 对后来者的建议

如果后续有人继续做这个产品或复用这套流程，建议遵守以下原则：

1. 先读 `product-design/` 下的约束文件，再做任何动作
2. 不要跳过产品经理验收和测试闸门
3. 不要因为产品简单就省略规则定义
4. 不要让任何角色同时承担“定义 + 实现 + 放行”三种职责
5. 修改时优先做最小正确改动
6. 所有阶段性提示词都应保留，以便后续溯源

## 12. 一句话总结

回本计算器项目证明了一件事：

**对于这类轻量产品，真正决定交付质量的，不是开发难度，而是能否建立一套清晰的约束、分工、流转、验证和闭环修正机制。**
