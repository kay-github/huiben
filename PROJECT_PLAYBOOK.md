# 回本计算器项目执行手册

- Project: 回本计算器
- Purpose: 将本项目沉淀为可直接复用的执行 SOP
- Scope: 适用于小功能、强约束、强调高质量交付的 AI 协作项目
- Last Updated: 2026-03-29

## 1. 这份手册解决什么问题

这份文档回答的不是“产品是什么”，而是“项目应该怎么做”。

它的目标是让后来者快速理解：

1. 这个项目的实现难度不高
2. 真正重要的是流程控制
3. 只要按这套 SOP 执行，就能稳定产出高质量结果

一句话概括：

**先定义，再约束，再执行，再验收，再修正，直到通过。**

## 2. 适用范围

这套手册适用于：

1. 单页面或少页面产品
2. 单功能或低复杂度产品
3. 技术实现不重，但体验要求高的项目
4. 需要多个 AI 角色协作的项目
5. 容易因为需求发散而失控的项目

## 3. 核心原则

### 3.1 文档先行

任何设计、开发、测试动作之前，必须先有约束文档。

### 3.2 角色隔离

1. 产品负责定义和验收
2. 设计负责输出视觉方案
3. 开发负责实现
4. 测试负责放行

不要让同一角色同时承担“定义 + 实现 + 放行”。

### 3.3 最小正确方案

如果有多种做法，优先选择：

1. 更小
2. 更稳
3. 更容易验证
4. 更不容易偏离产品边界

### 3.4 验收优先于感觉

“看起来差不多”不能算完成。只有经过明确验收和测试放行，才能算完成。

### 3.5 闭环修正

发现问题后不要重做一切，而是：

1. 指出问题
2. 锁定范围
3. 定向修正
4. 再验收

## 4. 角色职责

### 4.1 产品经理

负责：

1. 收敛需求
2. 定义边界
3. 输出 PRD、测试用例、开发规则
4. 审查设计
5. 审查开发结果
6. 决定是否进入下一阶段

### 4.2 UI 设计 AI

负责：

1. 基于边界设计界面
2. 按反馈定向迭代
3. 输出可供开发还原的定稿方向

### 4.3 开发 AI

负责：

1. 严格按文档实现
2. 处理定向修正
3. 在不同平台做受控转换

### 4.4 测试 AI

负责：

1. 按测试标准验证
2. 只输出通过 / 不通过
3. 发现问题时给出复现信息
4. 不改代码

### 4.5 人工决策者

负责：

1. 决定方向
2. 决定阶段切换
3. 决定是否上线或继续修正

## 5. 必备输入物

一个项目要进入稳定执行，至少要先具备这些输入物：

1. PRD
2. 测试用例
3. 开发规则
4. UI 提示词或设计说明
5. 设计定稿参考
6. 开发提示词
7. 测试提示词

如果缺这些输入物，就不要直接开始做代码。

## 6. 标准阶段 SOP

### 阶段 1：产品定义

目标：把模糊想法变成可执行范围。

动作：

1. 用一句话定义产品
2. 明确只做什么
3. 明确不做什么
4. 定义输入、输出、规则、异常状态
5. 写出核心卖点和视觉目标

输出：

1. PRD
2. 测试用例
3. 开发规则

通过条件：

1. 产品边界清晰
2. 公式和规则清晰
3. 验收标准清晰

### 阶段 2：设计收敛

目标：得到可开发的视觉方向。

动作：

1. 给 UI 设计 AI 明确提示词
2. 审查初稿是否越界
3. 如不合格，输出 V2、V3 定向修正提示词
4. 直到达到“可开发定稿”标准

输出：

1. UI 提示词迭代文件
2. 设计稿或设计导出代码

通过条件：

1. 无越界模块
2. 视觉重心正确
3. 可作为开发依据

### 阶段 3：开发实现

目标：在既定约束下完成最小正确实现。

动作：

1. 给开发 AI 明确必须遵守的文件
2. 明确冲突优先级
3. 明确实现目录和技术栈
4. 明确必须验证的测试点

输出：

1. 可运行代码
2. 开发说明或交付说明

通过条件：

1. 核心逻辑完成
2. 核心页面完成
3. 关键状态完成

### 阶段 4：产品经理验收

目标：判断实现是否符合产品定义。

动作：

1. 对照 PRD 审查
2. 对照测试用例检查
3. 标出当前真正影响验收的问题
4. 将问题转成定向修正提示词

输出：

1. 验收结论
2. 修正提示词（如果需要）

通过条件：

1. 没有中高优先级产品偏差

### 阶段 5：平台转换

目标：在不改产品本质的前提下，将已验收版本转换到另一个平台。

动作：

1. 以已验收版本作为行为基线
2. 在独立目录中实现新平台版本
3. 保持逻辑、文案、交互和视觉方向一致

输出：

1. 独立平台版本
2. 平台转换提示词

通过条件：

1. 新平台版本运行正常
2. 不改乱原版本
3. 不偏离已验收结果

### 阶段 6：测试闸门

目标：由独立测试角色决定是否放行。

动作：

1. 给测试 AI 测试提示词
2. 限制测试只输出通过 / 不通过
3. 不通过则问题回流开发 AI 修复
4. 修复后再次测试

输出：

1. 测试通过结论
2. 或测试问题列表

通过条件：

1. 测试 AI 明确写出“通过”

### 阶段 7：上线与沉淀

目标：把项目从一次性交付变成可复用经验。

动作：

1. 记录通过结论
2. 保留关键提示词
3. 写工作流文档
4. 写执行手册和文档索引

输出：

1. 工作流文档
2. Playbook
3. 索引文件

## 7. 阶段切换规则

任何阶段都不应该“差不多就往下走”。

必须满足：

1. 上一阶段有明确输出物
2. 输出物已被确认可用
3. 下一阶段收到的输入足够明确

否则宁可停在当前阶段，不要带着模糊问题继续推进。

## 8. 问题回流机制

当出现问题时，按以下方式处理：

1. 产品经理或测试 AI 指出问题
2. 问题必须具体到文件、行为或视觉偏差
3. 将问题转成定向修正提示词
4. 开发 AI 只修当前问题
5. 修完后重新验收或重测

这个机制的重点是：

1. 不大改
2. 不发散
3. 不重启项目

## 9. 提示词编写规则

一个可执行的提示词至少要包含：

1. 角色定位
2. 任务目标
3. 必须遵守的文件
4. 冲突优先级
5. 明确范围
6. 禁止事项
7. 验证要求
8. 输出格式

好的提示词不是“去做一下”，而是“告诉对方应该怎么做、不能怎么做、做到什么程度算完成”。

## 10. 完成定义

一个项目只有同时满足以下条件，才算真正完成：

1. 产品定义明确
2. 设计已收敛
3. 开发实现完成
4. 产品经理验收通过
5. 测试 AI 放行
6. 关键文档已归档
7. 后续接手者可以通过文档理解项目

## 11. 最小文档清单

对这类项目，建议至少保留以下文档：

1. PRD
2. 测试用例
3. 开发规则
4. UI 提示词
5. 开发提示词
6. 测试提示词
7. 工作流文档
8. 索引文档

## 12. 常见失败模式

### 12.1 先写代码，后补规则

结果：

1. 返工变多
2. 边界混乱
3. 不同角色理解不一致

### 12.2 设计自由发挥

结果：

1. 增加无关模块
2. 偏离产品本质
3. 增加开发返工

### 12.3 开发擅自扩展功能

结果：

1. 范围失控
2. 交付变慢
3. 验收标准失真

### 12.4 没有独立测试闸门

结果：

1. 开发自证正确
2. 问题后移到上线后才暴露

### 12.5 没有文档索引

结果：

1. 后来者不知道先看什么
2. 项目知识无法复用

## 13. 一套可复用的最简模板

如果后续做类似项目，可直接照这个模板走：

1. 写 `PRD`
2. 写 `测试用例`
3. 写 `开发规则`
4. 给 UI 设计 AI 出提示词
5. 做设计审查，直到可开发
6. 给开发 AI 出实现提示词
7. 做产品经理验收
8. 如有问题，给开发 AI 定向修正提示词
9. 必要时做平台转换
10. 给测试 AI 测试提示词
11. 不通过则修正并重测
12. 通过后写工作流文档和索引

## 14. 最终方法论

对这类项目来说，真正重要的不是“代码有多难”，而是是否建立了下面这条链路：

**产品定义 -> 规则固化 -> 角色分工 -> 定向执行 -> 独立验收 -> 闭环修正 -> 测试放行 -> 文档沉淀**

只要这条链路稳定，项目就会稳定。
