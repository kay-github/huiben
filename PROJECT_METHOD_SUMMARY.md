# 回本计算器项目方法论摘要

- Project: 回本计算器
- Purpose: 用一份短文档总结本项目可复用的方法论
- Last Updated: 2026-03-29

## 1. 一句话结论

这个项目真正的难点不在开发，而在于：

**如何用一套清晰的约束、分工、流转、验收和回流机制，把一个简单产品稳定地做对。**

## 2. 项目给出的核心启示

回本计算器是一个典型的“小功能、轻逻辑、强体验”项目。

它证明了一件事：

1. 技术实现简单，不代表项目管理简单
2. 简单产品最容易失控的地方，不是写不出来，而是边界漂移
3. 一旦缺少规则、角色分工和验收闸门，项目很容易出现：
   - 设计自由发挥
   - 开发擅自扩展
   - 测试标准漂移
   - 最终结果与最初目标不一致

因此，这类项目最有效的做法不是“直接开始做”，而是先建立一套受控流程。

## 3. 方法论核心

本项目沉淀出的核心方法论可以概括为这条链路：

**产品定义 -> 规则固化 -> 角色分工 -> 定向执行 -> 独立验收 -> 闭环修正 -> 测试放行 -> 文档沉淀**

这条链路的意义是：

1. 先把目标说清楚
2. 再把边界锁住
3. 再让不同角色各司其职
4. 每一轮都用验收结果推动下一轮，而不是靠主观感觉推进

## 4. 这套方法为什么有效

### 4.1 文档先行，避免理解漂移

先写 PRD、测试用例、开发规则，再做设计和开发。

作用：

1. 把模糊需求变成可执行边界
2. 让多个 AI 共享同一套判断标准
3. 让后续修正有明确依据

### 4.2 角色隔离，避免职责混乱

本项目把职责清晰拆成：

1. 产品经理：定义、约束、验收
2. UI 设计 AI：出稿、迭代、收敛
3. 开发 AI：实现、修正、平台转换
4. 测试 AI：验证、放行、不写代码

作用：

1. 防止一个角色同时定义、实现、放行
2. 防止“自己说自己是对的”

### 4.3 提示词化执行，保证可追溯

每一轮任务都通过文档化提示词下发，而不是口头指令。

作用：

1. 清楚说明目标、范围、禁止事项和输出格式
2. 便于复查
3. 便于后来者复用

### 4.4 多轮收敛，避免一次性豪赌

设计和开发都不是“一次做完”，而是：

1. 先出版本
2. 再审查偏差
3. 再定向修正
4. 再次验收

作用：

1. 问题更早暴露
2. 返工范围更小
3. 项目始终在可控轨道上推进

### 4.5 测试闸门，确保完成定义客观化

项目最终不是“感觉差不多了”就结束，而是要等测试 AI 明确给出：

1. 测试通过
2. 当前版本可视为可上线版本

作用：

1. 让完成有客观标准
2. 把质量控制从主观印象转成明确闸门

## 5. 标准工作方式

对于类似项目，推荐使用下面这套标准工作方式：

1. 先定义产品范围
2. 再写测试标准
3. 再写开发规则
4. 再做设计
5. 设计不合格就继续收敛
6. 设计通过后再开发
7. 开发完成后由产品经理做验收
8. 如不合格，给出定向修正提示词
9. 必要时再做平台转换版本
10. 最后交给测试 AI 放行
11. 测试不通过就修复并重测
12. 测试通过后沉淀文档和流程

## 6. 这套方法适合什么项目

尤其适合以下类型：

1. 单功能工具产品
2. 小程序、H5、官网单页、轻量 SaaS 页面
3. 实现不重，但对体验和收敛度要求高的项目
4. 多 AI 角色协作的项目

不一定适合：

1. 超大型复杂系统
2. 需求长期高速变化、边界本就不稳定的项目

## 7. 对后来者最重要的建议

如果你要复用这套方法，只记住下面三句话即可：

1. 不要跳过规则定义
2. 不要跳过独立验收和测试闸门
3. 不要把简单项目当成可以省略流程的项目

## 8. 最终摘要

回本计算器项目的方法论不是“如何开发一个计算器”，而是：

**如何把一个本来很容易被随手做坏的小项目，通过系统化的控制、分配、流转、验证和修正，稳定做成一个可交付、可上线、可复用的方法样板。**
